# 统一执行路线图

> 本文件汇总已确认的后续升级事项、依赖关系和实施门槛。`NEXT_VERSION_PLAN.md` 保留详细设计；本文件是执行排序的唯一入口。

## 执行规则

- 后续更新默认只做聚焦测试、完整 pytest 和 GUI 验收；除非用户明确要求，不打包 EXE，也不发布 GitHub。
- 每次功能范围新增、调整或合并后，先重新审核数据规模、调用频率、界面渲染、磁盘写入、后台线程和常驻进程，再决定优化方式和版本归属。
- 每个批量写入工作流必须遵循：校验 -> 一次备份 -> 一次保存 -> 一次部署或预览刷新 -> 一次结果提示。
- 受管 Rime 文件始终优先保持原生可用；本地增强数据不能污染词库格式。

## 已完成

- `v0.4.3`：自启后台常驻、重复启动唤醒、托盘恢复、原子写盘和可靠单文件更新交接。

## v0.4.4：运行底座、性能基线与发布架构

1. 延后加载设置页、符号表、系统词典索引；候选预览由轮询改为主动回传；大词典采用流式分批构建。
2. 增加仅开发可见的分段性能记录：冷启动、常驻内存、空闲 CPU、首次采集/编辑弹窗、候选预览响应。
3. 设计目录式发布包的清单、暂存、整目录切换与回滚协议；单文件 EXE 更新继续使用现有流程。只有明确要求时才实际打包。
4. 文档同步：更新 README 当前版本，并将 PRD 中已完成或不再适用的待确认项标记为历史状态。

## v0.5.0：数据安全与可追溯

前置：建立统一本地元数据存储及版本迁移策略。分组、显示拼音、历史、备注标签和分组规则必须有清晰的所有权、备份与恢复边界。

1. 词库健康检查：空文本、空编码、非法权重、重码、无效分组和可能无效的 Rime 配置；支持定位与确认后修复。
2. 恢复前差异预览：按受管文件显示新增、修改、删除；支持单文件恢复。
3. 持久化修改历史：新增、编辑、删除、批量分组和恢复均可追溯与恢复；采用追加记录与定期压缩。
4. 部署状态中心：显示最近部署结果、受影响文件、诊断信息和重新部署入口。
5. 外部文件协调：应用重新获得焦点或执行写入前检查受管文件签名；发现外部修改时提示刷新或对比，避免覆盖 Rime 或手工改动。

## v0.5.1：词库管理效率

1. 组合查询与筛选：编码、未分组、英文、低权重、最近修改等条件可组合。
2. 本地备注与标签：不写入 Rime；与统一元数据存储共用迁移、备份和恢复策略。
3. 自定义分组规则：预览后批量应用。
4. 批量操作收尾：批量删除或调整分组成功后，退出批量选择、清空选择并恢复工具栏。
5. 分组条目计数：标题后显示低饱和度数量，数据变化后同步刷新。
6. 重码集中管理：独立弹窗按重码文本或重码拼音聚合；二级窗口集中编辑另一侧项目。
7. 重码全量索引：基于全部受管词库、独立于当前搜索和分组；分别维护文本到编码、规范化编码到文本的可失效索引。
8. 性能边界：筛选和分组计数复用预聚合统计；重码和批量编辑使用模型视图或分页；索引只在词库变更或重载后重建。

## v0.6.0：导入、导出与智能决策

1. 导入、导出和迁移：流式读取、分批校验、冲突预览、一次事务式写入和一次部署。
2. 词条冲突决策：结合现有 Rime 候选预览、系统词典静态权重和自定义词库，帮助选择新增、覆盖或调整权重。

## 已审查的待确认项

- 方案受限配置 UI：SchemaRepo 已支持 `switches`、`custom_phrase` 和 key bindings 数据访问，但设置页当前只读显示方案信息。若恢复原始 PRD 目标，应新增安全的方案配置界面，并在写入前做差异预览和备份。
- 本地元数据存储形态：建议在 v0.5.0 开始前确定采用一个版本化 JSON 存储还是 SQLite。选择依据是历史记录规模、查询筛选需求和迁移复杂度；不得让历史、标签、规则分别创建互不兼容的 sidecar。
- 大型数据验收样本：在实现导入、历史与重码索引前，准备小、中、大三档测试词库，使用同一指标比较时间、内存和界面响应。

## 统一验收

- 每一项功能有聚焦单测，版本阶段运行完整 pytest。
- GUI 验收覆盖浅色、深色、水墨主题，普通启动、自启、托盘、热键、保存、部署与恢复相关路径。
- 批量与恢复操作必须验证：无遗漏、无重复写入、失败不破坏旧文件、通知与窗口内状态一致。
- 打包、GitHub 发布仅在用户明确要求时进行。
